# MSW Platform Rules

MSW engine's unique platform rules + per-map-type development guides. **Without knowing these rules, you will keep hitting the "code looks right but nothing works" trap.**

---

## 1. Rule Summary (Mandatory)

1. If **TileMapMode ‚ÜĒ Body mapping** is wrong, the entity will not move ‚ÄĒ without any error.
2. User scripts only work as `.mlua` + `.codeblock` **pairs**. Maker Refresh generates `.codeblock`.
3. If `SpriteRUID` is an empty string, the entity is **invisible on screen**.
4. When calling `SpawnByModelId`, passing anything other than a map entity to `parent` causes a runtime error.
5. Coordinates use **world units** (1 unit = 100 px). Pixel values will be off by 100x.
6. Maker only scans `RootDesk/`. User files placed in `Global/` will not be recognized.
7. Never modify `.d.mlua` or `.codeblock`.
8. CoreVersion is `26.5.0.0`. Do not work on mismatched versions.

## 3. File Pair Rules (File Pair System)

Every user script must exist as a `.mlua` + `.codeblock` pair.

| File | Role | Generation |
|------|------|------------|
| `ScriptName.mlua` | Lua source code | **Written directly by AI Agent** |
| `ScriptName.codeblock` | JSON metadata (type, inheritance, ID) | **Auto-generated by Maker Refresh** |

**Rules**:
- Creating only `.mlua` will not register the script with MSW ‚Üí **Maker Refresh is required**
- `.codeblock` is auto-generated by Maker analyzing the `script ... extends ...` declaration in `.mlua`
- Script location: `RootDesk/MyDesk/` directory
- After Maker Refresh, `Id` and `Name` in `.codeblock` are automatically aligned with the script name

### AI Workflow

```
1. AI writes .mlua script (RootDesk/MyDesk/)
2. User runs Refresh in Maker
3. Maker auto-generates .codeblock
4. Script is recognized ‚Üí ready to run
```

## 4. TileMapMode ‚ÜĒ Body Mapping (Most Important)

**Mismatched mapping causes the entity to not move. It fails silently with no error message.**

| Value | TileMapMode | View | Body Component | Map Component | Gravity | Terrain | Movement Axes | Collision System | Typical Genre |
|:---:|---|---|---|---|:---:|---|---|---|---|
| `0` | `MapleTile` | Side-view (side-scrolling) | `RigidbodyComponent` | `TileMapComponent` + `FootholdComponent` | O (built-in) | Foothold (line segments) | Left/Right + Jump (Y = gravity) | Foothold collision | Platformer, MapleStory style |
| `1` | `RectTile` | Top-down | `KinematicbodyComponent` | `RectTileMapComponent` | X | Square tile grid | Free 4-direction | Tile collision (Movable property) | Top-down RPG, Bomberman |
| `2` | `SideViewRectTile` | Side-view (side-scrolling) | `SideviewbodyComponent` | `RectTileMapComponent` | O (built-in) | Square tile grid | Left/Right + Jump (Y = gravity) | Tile collision (Collision property) | Side-view action, tile-based platformer |

> `.map` files serialize this field **as a number** (e.g. `"TileMapMode": 0`). The `Value` column above is the enum mapping.

**Check protocol** (mandatory preflight for every map-related task):
1. Identify the target map file (e.g. `./map/{mapname}.map`).
2. Open that `.map` file and read `MapComponent.TileMapMode` **as a number**. Because 0 / 1 / 2 decide the entire physics / movement / collision / terrain stack, **do not start any entity placement, model authoring, or script work without knowing this value**.
3. Confirm that Player.model / monster .model / every other dynamic entity carries the Body component from the matching row above.
4. On mismatch ‚Üí swap the Body component. This cannot be fixed by editing script code.

### LEA-3004 MissingComponent ‚ÄĒ the runtime signal of a TileMapMode ‚ÜĒ Body mismatch

If an entity has logic that requires one of the Body components (e.g. `MovementComponent`, `PlayerControllerComponent`, AI components) but the matching Body is not attached, the engine logs exactly one of the three messages below:

| TileMapMode | Required Body | Log when this Body is missing |
|---|---|---|
| `0` MapleTile | `RigidbodyComponent` | `[LEA-3004] MissingComponent : Entityžóź 'RigidbodyComponent'ÍįÄ žóÜžäĶŽčąŽč§.` |
| `1` RectTile | `KinematicbodyComponent` | `[LEA-3004] MissingComponent : Entityžóź 'KinematicbodyComponent'ÍįÄ žóÜžäĶŽčąŽč§.` |
| `2` SideViewRectTile | `SideviewbodyComponent` | `[LEA-3004] MissingComponent : Entityžóź 'SideviewbodyComponent'ÍįÄ žóÜžäĶŽčąŽč§.` |

**What it means**:
- The cause is almost always that a model whose Body does **not** match the map's `TileMapMode` was placed, or that a dynamic entity was used with **no Body at all**.
- **Fix**: add (or swap) the Body component in the matching row inside the `.model`, or in the map entity's `@components`, then `refresh`. **Do not** "fix" it by removing `MovementComponent` or bypassing the movement path ‚ÄĒ that only hides the root cause and breaks tile collision / enter-leave events.
- **Prevention**: following the Check protocol above prevents this error entirely. Missing-Body is a **silent-failure zone** that syntax / API validation cannot catch, so the moment you see `LEA-3004` in the logs, re-check the Body ‚ÜĒ TileMapMode mapping before anything else.

### TileMapMode switching is performed by the user in Maker

**Switching the mode itself between MapleTile ‚ÜĒ RectTile ‚ÜĒ SideViewRectTile** is not handled by AI editing the `.map` file manually. Mode switching involves swapping tile components, rebuilding Footholds, converting tile data formats, etc., all of which require Maker's internal processing ‚ÄĒ so **the user must change it directly in the Maker editor**.

**Exact Maker procedure to relay to the user (in Korean):**

1. Open the Maker editor's **Hierarchy** window.
2. **Right-click the target map entity** in the Hierarchy.
3. From the context menu, choose the **"Switch ..." option that matches the target mode** ‚ÄĒ i.e. Switch TileMap (MapleTile) / Switch RectTileMap (RectTile) / Switch SideViewRectTileMap (SideViewRectTile). Maker performs the conversion, swaps the tile component, and resets terrain as needed.
4. The user then saves; the AI runs MCP **`refresh`** and re-reads `MapComponent.TileMapMode` to confirm the new value.

> AI cannot do: change `TileMapMode` value by editing the `.map` JSON directly
> User action required: **Maker Hierarchy žįĹ ‚Üí ŪēīŽčĻ ŽßĶ žóĒŪčįŪčį žöįŪĀīŽ¶≠ ‚Üí "Switch ..." Ž©ĒŽČī žĄ†ŪÉĚ**
> AI can do: **recommend** the right mode for the user's game (matrix in [msw-general SKILL.md "Recommending the right mode"](../SKILL.md)); after the switch, fix Body component mismatches and adjust scripts

### DefaultPlayer's Body Component Structure

The DefaultPlayer model **contains all 3 Body components**.

```
DefaultPlayer
‚Ēú‚ĒÄ‚ĒÄ RigidbodyComponent      ‚Üź active on MapleTile maps
‚Ēú‚ĒÄ‚ĒÄ KinematicbodyComponent  ‚Üź active on RectTile maps
‚Ēú‚ĒÄ‚ĒÄ SideviewbodyComponent   ‚Üź active on SideViewRectTile maps
‚Ēú‚ĒÄ‚ĒÄ MovementComponent       ‚Üź high-level wrapper (always active)
‚ĒĒ‚ĒÄ‚ĒÄ PlayerControllerComponent ‚Üź input handling (always active)
```

**The engine automatically activates the appropriate Body component based on the map type.** The others remain inactive.

‚Üí When building custom monster/NPC models, only include the **Body component matching the map type**. You don't need all 3.

### Cautions When Switching Map Type

**A single map can use only one TileMapMode.**

When you change the map type:
1. **All existing terrain is reset** (irreversible).
2. Body components must be swapped (Player.model + every monster/NPC model).
3. Movement/collision scripts must be overhauled.
4. Event handlers must be replaced (Foothold events ‚ÜĒ RectTile events).

‚Üí **Lock in the map type early in the project and don't change it.**

## 5. Coordinate System and Screen

### Visible Screen Range

The camera is in 2D Orthographic mode. Centered on the player, the following region is visible:

| Platform | Width | Height | OrthographicSize | Example (camera at 0,0) |
|----------|-------|--------|:----------------:|------------------------|
| **PC** | **12.8** world units | **7.2** world units | 3.6 | X: ¬Ī6.4, Y: ¬Ī3.6 |
| **Mobile** | **9.6** world units | **5.4** world units | 2.7 | X: ¬Ī4.8, Y: ¬Ī2.7 |

- 1 world unit = **100 pixels** (PPU)
- Zoom range: **30% ~ 500%** (visible range varies with zoom)

> Use this range to judge whether entity placements are inside or outside the screen.

### World Coordinates

- Unit: **world units** (1 world unit = 100 pixels). Visible range is roughly ¬Ī6.4 units, so coordinate values are usually **single- to double-digit** in magnitude.
- Y axis: up is positive (standard game coordinate system).
- `TransformComponent.Position`: **local coordinate relative to the parent** (`Vector3`, world units). This is what gets serialized/synced.
- `TransformComponent.WorldPosition`: **absolute coordinate after walking the parent chain** (`Vector3`, world units). Mostly read-only; setting it internally converts back to Position via the parent's inverse matrix.
- For root entities with no parent, `Position == WorldPosition`.
- Z value: affects rendering order (larger = further back).
- Waypoints, spawn coordinates, movement targets ‚ÄĒ every Position-based value uses world units. Pixel values (in the hundreds) will be off by 100x.

### UI Coordinates

- **Reference resolution**: 1920 x 1080 (Landscape) / 1080 x 1920 (Portrait)
- Origin: screen center `(0, 0)`
- Screen edges: X `¬Ī960`, Y `¬Ī540` (Landscape)
- `UITransformComponent.anchoredPosition`: coordinate relative to parent
- Scaled against 1920x1080 regardless of actual device resolution

### Cautions

- World coordinates and UI coordinates are **separate spaces** ‚ÄĒ no direct conversion.
- A UI element's `Position` is fixed at `(0, 0, 0)` ‚ÄĒ **place via `anchoredPosition`**.
- To position UI elements at screen edges, use anchor presets (safer than directly setting anchoredPosition ¬Ī960/¬Ī540).

## 6. Map Layer Priority (Rendering Order)

An entity's display order is determined by these **3 priority levels**:

```
Priority 1: SpriteRendererComponent's SortingLayer (map layer)
Priority 2: SpriteRendererComponent's OrderInLayer
Priority 3: TransformComponent's Position Z value
```

### Default OrderInLayer by Model Type

When a model is placed on a map, a default OrderInLayer is auto-assigned based on its type:

| Model Type | Default OrderInLayer |
|------------|:--------------------:|
| Object | 0 |
| Tile | 1 |
| Monster, NPC, Platform, Ladder, Rope, Portal, Trap, Item | 2 |
| Other player avatars | 3 |
| My avatar | 4 |

> When OrderInLayer values match, **smaller Z values are drawn in front**. If an entity is hidden behind another, walk through these 3 priorities in order.

## 7. SpriteRUID Is Mandatory

If `SpriteRendererComponent`'s `SpriteRUID` is an empty string (`""`), the entity is **invisible on screen**.

- The component name alone does not reveal that RUID is required.
- All visible entities (player, monsters, objects) require a RUID.
- Default sprite RUID: `1705e3c5b2c146ac9a699f96fb067408` (placeholder when nothing else is available)

**Setting RUID**:
```lua
-- Set at runtime
self.Entity.SpriteRendererComponent.SpriteRUID = "ruid-string-here"

-- Set in .model file (Values array)
{
  "Name": "SpriteRUID",
  "Value": "1705e3c5b2c146ac9a699f96fb067408"
}
```

**Searching for RUIDs**: Use `_ResourceService` to look up assets. Only resources registered to the project are usable.

## 8. Entity Spawn Rules

### SpawnService Required Rules

```lua
-- Correct usage (model ID is case-insensitive)
_SpawnService:SpawnByModelId("chasemonster", "entityName", spawnPosition, mapEntity)

-- Error (parent is nil)
_SpawnService:SpawnByModelId("chasemonster", "entityName", spawnPosition, nil)  -- runtime error!
```

**Rules**:
- The first parameter `id` is the **model ID** ‚ÄĒ the string after `model://` in the `.model` file's EntryKey.
- Case-insensitive (`"ChaseMonster"`, `"chasemonster"` both work).
- If the model does not exist, returns nil (silent failure) ‚Üí **always nil-check the return value**.
- The `parent` parameter must receive a **map entity**.
- Passing `nil` produces a runtime error.
- Get the map entity via `self.Entity.CurrentMap`.

### SpawnByModelId vs SpawnByEntity

| Method | Purpose | Notes |
|--------|---------|-------|
| `SpawnByModelId(id, name, pos, parent)` | Create from a `.model` | Requires the model file |
| `SpawnByEntity(entity, name, pos, parent)` | Clone an existing entity | Requires the source entity |

### Spawn Initialization Order

The engine guarantees this execution order when an entity is created:

```
1. Empty entity created
2. Components in the Components array instantiated
3. Properties links wired up
4. Values array iterated ‚Üí defaults applied via TargetType + Name match
5. Synchronization (@Sync) registered
6. OnBeginPlay() callback invoked
```

**Implication**: At OnBeginPlay, every component and default value is already in place.

## 8.5 Dynamic Entities and Body Components

**Dynamic entity** = an entity that needs to move or collide with terrain at runtime.
Examples: Player, Monster, NPC, Pet/Follower, Projectile, Movable Trap, Pushable Object, Drop Item.

**Rule**: every dynamic entity needs a Body component, chosen by the map's `TileMapMode`:

| TileMapMode | Body component |
|-------------|----------------|
| `MapleTile` (0) | `RigidbodyComponent` |
| `RectTile` (1) | `KinematicbodyComponent` |
| `SideViewRectTile` (2) | `SideviewbodyComponent` |

Static decoration, trigger areas, VFX-only entities do **not** need a Body.

> ‚ö†ÔłŹ The three Body components are **not interchangeable** ‚ÄĒ APIs, properties, events, and physics models differ. Switching a map's mode requires rewriting movement/collision scripts per ¬ß9.x, not just swapping the component.

## 9. Per-Map-Type Development Guides

### 9.1 MapleTile (Side-View Platformer)

#### Grid Size

`TileMapComponent.GridSize` is a **fixed value `(0.45, 0.3)`** (`static readonly`). Cannot be changed.

#### Physics System

A Foothold-based physics system unique to MapleStory.

- **Gravity**: `RigidbodyComponent.Gravity` (has a default, adjustable)
- **Walking on footholds**: controlled via `WalkSpeed`, `WalkAcceleration`, `WalkDrag`
- **Air movement**: controlled via `AirAccelerationX`, `AirDecelerationX`, `FallSpeedMaxX/Y`
- **Jump**: `WalkJump` (height), `JumpBias` (air time)
- **Mass**: `Mass` (affects acceleration/deceleration responsiveness)

```lua
-- Example default setup for key RigidbodyComponent properties
local rb = self.Entity.RigidbodyComponent
rb.Gravity = 30          -- gravity strength
rb.WalkSpeed = 3         -- max walk speed
rb.WalkJump = 6          -- jump height
rb.WalkAcceleration = 10 -- walk acceleration
rb.WalkDrag = 1          -- walk friction
rb.Mass = 1              -- mass
```

#### Terrain and Collision

- **FootholdComponent**: manages all footholds on the map. Interacts with `RigidbodyComponent`.
- Footholds are made of **line segments (StartPoint~EndPoint)**.
- Walking only happens on footholds; without a foothold, the entity **falls under gravity**.
- `DownJump()`: jump downward (drops through a foothold).
- `IsOnGround()`: check if standing on a foothold.
- `GetCurrentFoothold()`: info about the foothold currently underfoot.
- `PredictFootholdEnd(distance, isForward)`: predict distance to the end of the foothold.

```lua
-- Check if on a foothold
if self.Entity.RigidbodyComponent:IsOnGround() then
    -- logic that runs only on a foothold
end

-- Detect foothold end (for AI monsters)
if self.Entity.RigidbodyComponent:PredictFootholdEnd(1, true) then
    -- within 1 unit of the right edge ‚Üí turn around
end
```

#### Events

| Event | When |
|-------|------|
| `FootholdEnterEvent` | Landed on a foothold |
| `FootholdLeaveEvent` | Left a foothold |
| `FootholdCollisionEvent` | Collided with a foothold |
| `RigidbodyAttachEvent` | Entity attached via AttachTo |
| `RigidbodyDetachEvent` | Detached via Detach |

#### Monster/NPC Development

**Required**:
1. The monster `.model` must include `RigidbodyComponent`.
2. If `Gravity` is 0, the monster **floats in the air** ‚Üí always set a positive value.
3. If `WalkSpeed` is 0, **movement is impossible**.
4. Spawn Y must be above a foothold so the monster falls and lands (below a foothold = infinite falling).

```lua
-- MapleTile monster basic patrol pattern
@Component
script MonsterAI extends Component

property boolean movingRight = true

[server only]
void OnUpdate(number delta)
{
    local rb = self.Entity.RigidbodyComponent
    if rb == nil then return end

    -- Only move while on a foothold
    if rb:IsOnGround() == false then return end

    -- Detect foothold end ‚Üí flip direction
    if rb:PredictFootholdEnd(0.5, self.movingRight) then
        self.movingRight = not self.movingRight
    end

    -- Set movement direction
    local dir = 1
    if self.movingRight == false then dir = -1 end
    rb.MoveVelocity = Vector2(dir, 0)
}

end
```

#### Special Features

- **KinematicMove mode**: setting `RigidbodyComponent.KinematicMove = true` switches to top-down movement (top-down movement on a MapleTile map).
- **AttachTo/Detach**: attach an entity to another (moving platforms, etc.).
- **AddForce/SetForce**: apply physics-based forces (knockback, push).

### 9.2 RectTile (Top-Down)

#### Physics System

**No** gravity. Free 4-direction movement. Tile-based collision.

- **Movement speed**: `KinematicbodyComponent.SpeedFactor` (separate X and Y multipliers)
- **Jump**: optional (`EnableJump = true` produces a visual-only jump, no actual height change)
- **Shadow**: `EnableShadow` shows a shadow during the jump

```lua
-- Key KinematicbodyComponent properties
local kb = self.Entity.KinematicbodyComponent
kb.SpeedFactor = Vector2(3, 3)     -- movement speed multiplier
kb.EnableJump = true                -- enable jump
kb.JumpSpeed = 5                    -- jump speed
kb.JumpDrag = 3                     -- fall speed
kb.EnableTileCollision = true       -- enable tile collision
kb.EnableShadow = true              -- show shadow
kb.ShadowSize = Vector2(0.5, 0.2)  -- shadow size
```

#### Terrain and Collision

- **RectTileMapComponent**: square tile grid.
- Per-tile **Movable property**: configure passable/blocked (in the tile editor).
- `EnableTileCollision`: toggle collision detection.
- Coordinate conversion: `ToCellPosition(worldPos)` ‚ÜĒ `ToWorldPosition(cellPos)`

```lua
-- World coords ‚Üí tile coords
local tilemap = self.Entity.CurrentMap:GetFirstChildComponentByTypeName("RectTileMapComponent")
local cellPos = tilemap:ToCellPosition(self.Entity.TransformComponent.WorldPosition)
local tileInfo = tilemap:GetTile(cellPos)

if tileInfo ~= nil then
    log("Current tile: " .. tileInfo.Name)
end
```

#### Events

| Event | When |
|-------|------|
| `RectTileEnterEvent` | Entered a tile |
| `RectTileLeaveEvent` | Left a tile |
| `RectTileCollisionBeginEvent` | Started touching a non-passable tile |
| `RectTileCollisionEndEvent` | Stopped touching a non-passable tile |
| `KinematicbodyJumpEvent` | Jump state changed |

#### Monster/NPC Development

**Required**:
1. Include `KinematicbodyComponent` in the monster `.model`.
2. **There is no gravity**, so no fall handling is needed.
3. Free movement in all 4 directions.
4. If `SpeedFactor` is `(0,0)`, movement is impossible.

```lua
-- RectTile monster basic patrol pattern (top-down)
@Component
script MonsterPatrol extends Component

@Sync
property int32 patrolIndex = 0

property table patrolPoints = {
    Vector2(1, 0), Vector2(0, 1), Vector2(-1, 0), Vector2(0, -1)
}

[server only]
void OnUpdate(number delta)
{
    local kb = self.Entity.KinematicbodyComponent
    if kb == nil then return end

    local dir = self.patrolPoints[(self.patrolIndex % #self.patrolPoints) + 1]
    kb.MoveVelocity = dir
}

end
```

#### Special Features

- **Tile-based speed change**: use `RectTileEnterEvent` to adjust speed when entering specific tiles.
- **Dynamic tile placement**: `SetTile()`, `RemoveTile()`, `BoxFill()` to modify the map at runtime.
- **Tile-name-based logic**: distinguish tile types using `tileInfo.Name`.

### 9.3 SideViewRectTile (Side-View Tile)

#### Physics System

A mode that combines RectTile's tile grid with side-view gravity.

- **Gravity**: built into the engine (no separate Gravity property; tune fall speed via `JumpDrag`).
- **Movement**: left/right + jump.
- **Jump**: `JumpSpeed` (jump speed), `JumpDrag` (fall speed).
- **Down jump**: `EnableDownJump`, `DownJumpSpeed`.

```lua
-- Key SideviewbodyComponent properties
local svb = self.Entity.SideviewbodyComponent
svb.JumpSpeed = 5         -- jump speed (higher = jumps higher)
svb.JumpDrag = 3          -- fall speed (higher = falls faster)
svb.EnableDownJump = true  -- enable down jump
svb.DownJumpSpeed = 3.3    -- down jump rebound speed
```

#### Terrain and Collision

- Uses **RectTileMapComponent** (same tile system as RectTile).
- Concept of standing on tiles + falling under gravity.
- `IsOnGround()`: standing on a tile or not.
- `GetUnderfootTile()`: info about the tile currently underfoot.

```lua
-- Check the tile currently underfoot
local svb = self.Entity.SideviewbodyComponent
local tileInfo = svb:GetUnderfootTile()
if tileInfo ~= nil then
    log("Underfoot tile: " .. tileInfo.Name)
end
```

#### Events

| Event | When |
|-------|------|
| `RectTileEnterEvent` | Entered a tile |
| `RectTileLeaveEvent` | Left a tile |
| `RectTileCollisionBeginEvent` | Started touching a collision tile (wall detection) |
| `RectTileCollisionEndEvent` | Stopped touching a collision tile |

#### Monster/NPC Development

**Required**:
1. Include `SideviewbodyComponent` in the monster `.model`.
2. **Gravity is auto-applied** ‚Üí spawn on top of a tile to avoid falling.
3. Only left/right movement (no top-down movement).
4. Drive movement with `MoveVelocity`.

```lua
-- SideViewRectTile monster basic patrol pattern
@Component
script MonsterWalk extends Component

@Sync
property boolean movingRight = true

[server only]
void OnUpdate(number delta)
{
    local svb = self.Entity.SideviewbodyComponent
    if svb == nil then return end

    -- Only move while on the ground
    if svb:IsOnGround() == false then return end

    -- Set movement direction
    local dir = 1
    if self.movingRight == false then dir = -1 end
    svb.MoveVelocity = Vector2(dir, 0)
}

-- Flip direction on wall collision
[self]
HandleRectTileCollisionBeginEvent(RectTileCollisionBeginEvent event)
{
    local normal = event.Normal
    if normal == Vector2.left or normal == Vector2.right then
        self.movingRight = not self.movingRight
    end
}

end
```

#### Special Features

- **Wall detection**: identify wall direction via the `Normal` vector on `RectTileCollisionBeginEvent`.
- **Custom movement**: drive `MoveVelocity` directly to implement slippery floors, accel/decel.
- **Dynamic tiles**: same runtime tile manipulation as RectTile.

## 10. MovementComponent ‚ÄĒ Common Wrapper

`MovementComponent` is a **high-level movement controller** usable with all 3 Body components.

```lua
-- MovementComponent (works regardless of map type)
local movement = self.Entity.MovementComponent
movement.InputSpeed = 3       -- movement speed
movement.JumpForce = 1.5      -- jump strength
movement:Jump()               -- perform jump
movement:DownJump()           -- down jump
movement:MoveToDirection(Vector2(1, 0), delta)  -- directional movement
movement:Stop()               -- stop
```

**How InputSpeed becomes actual velocity per map type** (engine-internal, `MovementComponentPartDefault`):

| TileMapMode | Actual velocity | Notes |
|---|---|---|
| **MapleTile** | `InputSpeed` is passed straight to the Rigidbody | ‚ÄĒ |
| **RectTile** | `direction * InputSpeed / 1.2f` | Default 1.2 for migration compatibility |
| **SideViewRectTile** | `direction.x * InputSpeed * 1.5f`, Y keeps existing velocity | Speed correction similar to Rigidbody |

- `InputSpeed` default: `1.0` (`MovementComponent`'s `[MODProperty]`, `@Sync`)
- The same `InputSpeed = 3` will feel different across map types.

### Interplay with PlayerControllerComponent

`PlayerControllerComponent` handles input ‚Üí action mapping and internally uses `MovementComponent`.

- Default key mapping: arrow keys (movement), Alt/Space (jump), down + jump (down jump).
- For custom movement: set `PlayerControllerComponent.Enable = false` and drive the Body component directly.

## 11. Troubleshooting ‚ÄĒ Common Mistakes

### Map Type Mismatch

| Symptom | Cause | Fix |
|---------|-------|-----|
| Entity does not move (no error) | Body component ‚ÜĒ TileMapMode mismatch | Check the ¬ß4 mapping table and swap the Body |
| `[LEA-3004] MissingComponent : Entityžóź 'RigidbodyComponent'ÍįÄ žóÜžäĶŽčąŽč§.` | A dynamic entity without `RigidbodyComponent` placed on a MapleTile map (`TileMapMode = 0`) | Add `RigidbodyComponent` to the model / entity per the ¬ß4 mapping table |
| `[LEA-3004] MissingComponent : Entityžóź 'KinematicbodyComponent'ÍįÄ žóÜžäĶŽčąŽč§.` | A dynamic entity without `KinematicbodyComponent` placed on a RectTile map (`TileMapMode = 1`) | Add `KinematicbodyComponent` to the model / entity per the ¬ß4 mapping table |
| `[LEA-3004] MissingComponent : Entityžóź 'SideviewbodyComponent'ÍįÄ žóÜžäĶŽčąŽč§.` | A dynamic entity without `SideviewbodyComponent` placed on a SideViewRectTile map (`TileMapMode = 2`) | Add `SideviewbodyComponent` to the model / entity per the ¬ß4 mapping table |
| Monster floats in mid-air | `Gravity = 0` on MapleTile | Set `Gravity` to a positive value |
| Monster falls off the foothold edge | No foothold-end handling on MapleTile | Use `PredictFootholdEnd` to flip direction |
| Monster disappears off the map | Gravity code added on RectTile | Remove gravity-related code |
| RectTile jump cannot clear walls | RectTile jump is **visual only** | Change the tile's Movable property |
| Floats in mid-air on SideViewRectTile | Using `KinematicbodyComponent` | Replace with `SideviewbodyComponent` |
| Tile collision broken on SideViewRectTile | Using `RigidbodyComponent` | Replace with `SideviewbodyComponent` |
| `PredictFootholdEnd` errors on SideViewRectTile | Not a Foothold system (MapleTile only) | Detect walls via `RectTileCollisionBeginEvent` |
| Monster gets stuck against a wall | No wall detection logic on SideViewRectTile | Use `RectTileCollisionBeginEvent` + `Normal` |

### Other Common Pitfalls

| Symptom | Cause | Fix |
|---------|-------|-----|
| Entity invisible on screen | `SpriteRUID = ""` | Look up RUID via the Resource API and set it |
| Entity moves only via `TransformComponent` | Gravity/collision ignored | Use `MoveVelocity` through the Body component |
| Movement only on the client, no sync | `[server only]` not used | Run movement on the server |
| `SpawnByModelId` runtime error | `parent = nil` | Pass `self.Entity.CurrentMap` |
| `SpawnByModelId` returns nil | Wrong/missing model ID | Always nil-check the return value |
| Coordinate values off by 100x | Pixel values used | Convert to world units (√∑100) |
| File not visible in Maker | Created under `Global/` | Move to `RootDesk/MyDesk/` |

## 14. ID Generation Rules

ID rules used by AI Agent when creating/editing files.

### EntryKey Patterns

| File Type | EntryKey Format | Example |
|-----------|-----------------|---------|
| `.model` | `model://{name_lowercase}` | `model://chasemonster` |
| `.map` | `map://{mapname_lowercase}` | `map://map01` |
| `.ui` | `ui://{root_entity_UUID}` | `ui://c1ee38e4-b61d-4299-a1dd-d7a9b11f55c5` |
| `.codeblock` | `codeblock://{scriptname_lowercase}` | `codeblock://monster` |
| `.config` | `config://{fixed_value}` | `config://world`, `config://sectors` |

### Entity ID

Entity IDs inside `.ui` and `.map` files use **UUID v4** (with hyphens).

```
Format: xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
Example: bdadf19a-cc27-4a45-99c6-7a439c858a1b
```

- Generate a new UUID whenever AI adds an entity.
- A `.ui` file's EntryKey must match the root entity's UUID.

### Model ID

- `.model`'s `ContentProto.Json.Name`: display name (e.g., `"ChaseMonster"`).
- The identifier in EntryKey: lowercase of the name (e.g., `model://chasemonster`).
- This is the value passed as the first parameter to `SpawnByModelId()`.

### Top-level Id / GameId

- For every file, top-level `Id` and `GameId`: **always keep as empty string** `""`.
- These are populated by the system at runtime; AI must not touch them.

### CollisionGroup ID

- System groups: `MOD@{name}` format (do not modify).
- Custom groups: 32-char hex (no hyphens), e.g., `a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6`.
